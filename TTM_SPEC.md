# Спецификация веб-приложения TTM

**Продукт:** TTM  
**Компания:** Транстелематика  
**Тип:** суверенная система управления задачами, планами и командной аналитикой  
**Формат документа:** техническая и продуктовая спецификация для нейросети-разработчика  
**Язык интерфейса:** русский  
**Основной стек:** Django  
**AI-модель:** Gemini 3.1 Flash Lite  
**Важное ограничение:** Docker не использовать и не требовать для запуска.

---

## 1. Главная задача для нейросети

Создай работающий прототип корпоративного веб-приложения **TTM** для компании **«Транстелематика»**.

Приложение должно помогать сотрудникам, руководителям команд и руководителям подразделений управлять задачами на разных горизонтах планирования: **год, квартал, месяц, неделя**. Это не должен быть простой список задач. Интерфейс должен выглядеть как полноценный внутренний продукт крупной технологической компании: чистый, строгий, надёжный, аналитический, с понятными дашбордами и AI-помощником.

Нужно реализовать прототип, который можно показать на хакатоне: с демонстрационными данными, рабочими страницами, фильтрами, формами, канбаном, календарём, аналитикой, рисками, ролями и AI-сценариями.

---

## 2. Ключевая идея продукта

TTM решает проблему разрозненного управления задачами. Сейчас задачи могут храниться в таблицах, почте, мессенджерах, заметках и внешних SaaS-сервисах. Из-за этого руководителям сложно быстро понять:

- какие задачи сейчас в работе;
- какие задачи просрочены;
- какие задачи находятся в зоне риска;
- кто из сотрудников перегружен;
- какие отделы выполняют планы в срок;
- где требуется управленческое внимание;
- какие действия нужно предпринять на этой неделе.

TTM должна давать единую картину по задачам, срокам, статусам, загрузке сотрудников, просрочкам, рискам и выполнению планов.

## 3. Технологический стек

### 3.1. Обязательный стек

Используй Django как основной фреймворк.

Рекомендуемый стек:

- **Backend:** Django 5.x или актуальная стабильная версия Django.
- **Frontend:** Django Templates + Tailwind CSS + небольшой JavaScript там, где это действительно нужно.
- **База данных для демо:** SQLite, чтобы прототип запускался быстро без сложной настройки.
- **База данных для корпоративного контура:** PostgreSQL, предусмотреть возможность переключения через переменные окружения.
- **Авторизация:** встроенная авторизация Django + группы/роли.
- **Графики:** Chart.js, Apache ECharts или аналогичная лёгкая библиотека.
- **Канбан drag-and-drop:** SortableJS или нативный JavaScript.
- **Календарь:** FullCalendar или простой кастомный календарь на Django Templates.
- **AI:** отдельный сервисный слой `ai_service`, подключающий Gemini 3.1 Flash Lite.
- **Стили:** Tailwind CSS, можно через локальную сборку или подключение для демо; архитектурно предусмотреть локальное хранение статических файлов.

### 3.2. Что не использовать

- Не использовать Docker.
- Не делать обязательную зависимость от React, Next.js или Node.js.
- Не строить прототип как SPA, если это усложняет реализацию.
- Не делать интерфейс в виде простой таблицы без визуальной аналитики.
- Не отправлять в AI весь объём корпоративных данных без фильтрации и минимизации.

### 3.3. Разрешено использовать точечно

Можно использовать JavaScript-библиотеки для интерактивных элементов:

- диаграммы;
- drag-and-drop в канбане;
- календарь;
- модальные окна;
- быстрые фильтры;
- динамическое обновление статуса.

Django должен оставаться основой приложения.

## 4. Запуск проекта без Docker

Нейросеть должна создать проект, который запускается локально так:

```bash
python -m venv .venv
source .venv/bin/activate  # Linux / macOS
# или
.venv\Scripts\activate     # Windows

pip install -r requirements.txt
python manage.py migrate
python manage.py seed_demo_data
python manage.py runserver
```

Если используется Tailwind-сборка, нужно дать простую инструкцию без Docker. Например:

```bash
python manage.py tailwind install
python manage.py tailwind start
```

или использовать уже подготовленные CSS-файлы в `static/`.

В проекте должны быть:

- `README.md` с запуском;
- `.env.example`;
- `requirements.txt`;
- команда для генерации демо-данных;
- суперпользователь или демо-пользователи с паролями в README.

## 5. Нефункциональные требования

### 5.1. Язык

Весь интерфейс, кнопки, сообщения, статусы, подсказки, пустые состояния, ошибки и тексты AI должны быть на русском языке.

### 5.2. Визуальный стиль

Стиль: корпоративный, технологичный, строгий, чистый, надёжный.

Интерфейс должен быть:

- аккуратным;
- без визуального шума;
- с большим количеством воздуха;
- с ясной иерархией;
- с bento-style карточками;
- с мягкими скруглениями 16–24 px;
- с аккуратными тенями;
- с понятной аналитической структурой.

### 5.3. Адаптивность

Приложение должно корректно работать на:

- ноутбуке;
- широком мониторе;
- планшете;
- мобильном экране в базовом режиме.

Основной сценарий демонстрации — desktop.

### 5.4. Производительность

Для демо-данных приложение должно работать быстро без тяжёлой клиентской логики. Основные страницы должны загружаться без заметных задержек.

## 6. Фирменный стиль

### 6.1. Цвета

Используй палитру Транстелематики:

| Название | HEX | Использование |
|---|---:|---|
| Белый | `#FFFFFF` | основной фон карточек |
| Светло-серый | `#F7F7F7` | общий фон приложения |
| Глубокий синий | `#020F52` | важные заголовки, тёмные акценты |
| Спокойный синий | `#003C97` | основной корпоративный акцент |
| Цифровой синий | `#269AE6` | вторичный акцент, графики, ссылки |
| Светло-синий | `#9CDDE6` | лёгкие подложки, выделения |

Фирменный градиент:

```css
linear-gradient(135deg, #020F52 0%, #003C97 42%, #269AE6 76%, #9CDDE6 100%)
```

Градиент использовать умеренно:

- в hero-блоках;
- в KPI-карточках;
- в боковом акценте;
- в AI-инсайтах;
- в крупных заголовках или декоративных линиях.

### 6.2. Типографика

Основной шрифт:

1. `Golos Text`, если доступен;
2. fallback: `Inter`, `system-ui`, `sans-serif`.

Начертания:

- заголовки — SemiBold;
- основной текст — Regular;
- числовые показатели — SemiBold.

### 6.3. Тон коммуникации

Тексты должны быть:

- простыми, но точными;
- без маркетингового шума;
- уверенными, но без пафоса;
- дружелюбными, но не фамильярными;
- ориентированными на надёжность, контроль, ответственность и результат.

Примеры микрокопи:

- «Пока задач нет. Создайте первую задачу или выберите другой фильтр.»
- «Задача просрочена. Рекомендуется обновить статус или назначить новый срок.»
- «Задача находится в зоне риска. AI-помощник рекомендует обратить на неё внимание.»
- «У сотрудника высокая загрузка на текущую неделю.»
- «Задача создана»
- «Статус обновлён»
- «Комментарий добавлен»
- «Отчёт сформирован»

## 7. Роли пользователей

В системе должно быть 4 роли.

### 7.1. Сотрудник

Возможности:

- видит свои задачи;
- видит дедлайны;
- понимает приоритеты;
- меняет статус своих задач;
- оставляет комментарии;
- видит историю по своим задачам;
- получает уведомления о приближении дедлайна;
- открывает карточку задачи и смотрит детали.

Ограничения:

- не видит все задачи компании;
- не управляет пользователями;
- не изменяет чужие задачи, кроме комментариев, если это разрешено.

### 7.2. Руководитель команды

Возможности:

- создаёт задачи;
- назначает ответственных;
- редактирует задачи своей команды;
- контролирует выполнение задач команды;
- видит просроченные и рискованные задачи;
- анализирует загрузку сотрудников;
- получает краткую сводку по команде;
- фильтрует задачи по сотруднику, статусу, периоду и дедлайну;
- использует AI-помощника для анализа ситуации.

### 7.3. Руководитель подразделения

Возможности:

- видит аналитику по отделам своего подразделения;
- контролирует выполнение планов за год, квартал, месяц и неделю;
- анализирует просрочки;
- видит распределение задач по сотрудникам;
- сравнивает отделы;
- получает управленческие подсказки от AI-помощника;
- формирует отчёты.

### 7.4. Администратор

Возможности:

- управляет пользователями;
- управляет отделами;
- управляет ролями;
- видит все данные;
- управляет настройками системы;
- видит аудит действий пользователей.

## 11. Логика расчёта риска

Риск задачи рассчитывается автоматически. Расчёт должен происходить:

- при создании задачи;
- при редактировании задачи;
- при изменении статуса;
- при открытии дашбордов;
- при запуске команды пересчёта рисков.

### 11.1. Факторы риска

Добавляй баллы риска:

| Фактор | Баллы |
|---|---:|
| Дедлайн уже прошёл, а задача не выполнена | +50 |
| До дедлайна меньше 3 дней, задача не выполнена | +25 |
| Приоритет высокий | +15 |
| Приоритет критический | +25 |
| Статус не менялся больше 7 дней | +15 |
| Ответственный перегружен | +20 |
| Задача на согласовании больше 5 дней | +20 |
| Прогресс меньше 30%, а до дедлайна меньше 5 дней | +20 |
| Нет комментариев и прогресса | +10 |
| Задача связана с годовой или квартальной целью | +10 |
| Задача без ответственного | +30 |
| Задача без дедлайна | +15 |

### 11.2. Уровни риска

| Баллы | Риск |
|---:|---|
| 0–24 | Низкий |
| 25–49 | Средний |
| 50–79 | Высокий |
| 80+ | Критический |

### 11.3. Пример текста риска

`Высокий риск: дедлайн через 2 дня, статус «В работе», у ответственного 8 активных задач.`

Для каждой рискованной задачи показывай:

- уровень риска;
- причину риска;
- рекомендацию;
- цветовой индикатор.

## 12. Главная страница

Главная страница — рабочий командный обзор. Руководитель должен за 30 секунд понять, где нужны действия.

### 12.1. Приветственный блок

Содержимое:

- «Добрый день, Иван»;
- «Сегодня важно обратить внимание на 6 задач»;
- краткая AI-сводка дня;
- фильтр периода.

Пример AI-сводки:

> На этой неделе у команды 34 задачи. 18 находятся в работе, 7 выполнены, 5 имеют высокий приоритет, 4 находятся в зоне риска просрочки.

### 12.2. KPI-карточки

Показать карточки:

- Всего задач;
- В работе;
- Выполнено;
- Просрочено;
- В зоне риска;
- Высокий приоритет;
- Средняя загрузка команды;
- Выполнение плана за период.

Карточки должны быть в bento-style сетке.

### 12.3. Блок «Требуют внимания»

Показывать задачи, которые:

- просрочены;
- близки к дедлайну;
- имеют высокий или критический приоритет;
- долго не меняли статус;
- находятся на согласовании слишком долго;
- закреплены за перегруженными сотрудниками.

### 12.4. Мини-дашборд

Показать:

- распределение задач по статусам;
- выполнение по периодам;
- нагрузку по отделам.

### 12.5. Блок AI-помощника

Показать короткий инсайт:

> В отделе HR 4 задачи находятся в риске просрочки. Основное внимание стоит обратить на подготовку хакатона и согласование документов. У Иванова самая высокая загрузка на текущую неделю.

Кнопки:

- «Открыть AI-помощника»;
- «Показать рискованные задачи»;
- «Сформировать отчёт».

### 12.6. Ближайшие дедлайны

Список задач на ближайшие 7 дней с цветовой индикацией срочности.

## 21. AI-помощник

AI-помощник должен выглядеть как корпоративный аналитический ассистент, а не декоративный чат.

### 21.1. Техническая архитектура AI

Создать сервисный слой:

```text
ttm/ai_service/
  __init__.py
  providers.py
  prompts.py
  schemas.py
  analysis.py
```

В `.env.example`:

```env
GEMINI_API_KEY=
GEMINI_MODEL=gemini-3.1-flash-lite
AI_PROVIDER=gemini
AI_MOCK_MODE=true
```

Если `GEMINI_API_KEY` отсутствует, приложение должно работать в mock-режиме и возвращать заранее подготовленные реалистичные AI-сводки. Это важно для демонстрации на хакатоне.

### 21.2. Минимизация данных

Перед отправкой данных в AI:

- не отправлять лишние персональные данные;
- отправлять только нужный срез задач;
- для аналитики использовать агрегаты;
- не отправлять вложения;
- не отправлять пароли, токены, внутренние настройки;
- логировать факт обращения к AI без сохранения полного промпта, если это может содержать корпоративные данные.

### 21.3. Возможности AI-помощника

#### 21.3.1. Краткая сводка по задачам команды

Пример:

> На этой неделе у команды 34 задачи. 18 находятся в работе, 7 выполнены, 5 имеют высокий приоритет, 4 находятся в риске просрочки.

#### 21.3.2. Выявление задач с риском просрочки

AI анализирует:

- дедлайн;
- текущий статус;
- приоритет;
- длительность без изменений;
- загрузку ответственного;
- наличие комментариев;
- период планирования.

#### 21.3.3. Подсказки руководителю

AI должен отвечать:

- на какие задачи обратить внимание;
- кому нужна помощь;
- где есть перегрузка;
- какие дедлайны под угрозой;
- какие задачи долго не двигаются.

#### 21.3.4. Анализ перегрузки сотрудников

AI показывает сотрудников, у которых:

- слишком много задач;
- много задач высокого приоритета;
- есть просроченные задачи;
- несколько дедлайнов в ближайшие дни.

#### 21.3.5. Генерация отчёта за неделю

AI создаёт текст:

- что выполнено;
- что в работе;
- что просрочено;
- какие риски;
- какие рекомендации;
- что нужно сделать на следующей неделе.

#### 21.3.6. Создание задачи из текстового описания

Пользователь пишет:

> Нужно до пятницы подготовить отчёт по затратам подразделения, ответственная Елена, отдел Финансы, высокий приоритет.

AI должен предложить заполненную форму задачи.

Результат должен быть структурированным JSON:

```json
{
  "title": "Подготовить отчёт по затратам подразделения",
  "description": "Подготовить отчёт по затратам подразделения до пятницы.",
  "responsible": "Елена",
  "department": "Финансы",
  "priority": "high",
  "status": "planned",
  "deadline_text": "до пятницы",
  "planning_period": "week"
}
```

#### 21.3.7. Ответы на вопросы по текущим задачам

Примеры вопросов:

- «Какие задачи просрочены в HR?»
- «Кто перегружен на этой неделе?»
- «Какие задачи высокого приоритета требуют внимания?»
- «Что нужно сделать до конца недели?»
- «Какие квартальные задачи отстают?»

### 21.4. Интерфейс AI-помощника

На странице должны быть:

- чат;
- готовые быстрые вопросы;
- карточки инсайтов;
- список рекомендаций;
- кнопка «Сформировать отчёт»;
- кнопка «Создать задачу из текста»;
- кнопка «Показать рискованные задачи».

### 21.5. Тон AI

AI должен быть:

- деловым;
- точным;
- без лишних эмоций;
- с понятными рекомендациями;
- без обещаний невозможного;
- с объяснением причин выводов.

### 21.6. Системный промпт для AI-помощника

Используй как основу:

```text
Ты корпоративный аналитический ассистент TTM для компании «Транстелематика».
Отвечай на русском языке. Тон: деловой, точный, спокойный, без маркетингового шума.
Твоя задача — помогать руководителям и сотрудникам понимать состояние задач, риски, просрочки, загрузку людей и выполнение планов.
Не обещай невозможного. Всегда объясняй, на основании каких факторов сделан вывод.
Если данных недостаточно, прямо скажи, каких данных не хватает.
Не раскрывай технические токены, настройки системы и конфиденциальные данные.
Давай конкретные управленческие рекомендации.
```

```python
class BaseAIProvider:
```

- `GeminiProvider`;
- `MockAIProvider`;
- в будущем `LocalLLMProvider`.

## 32. Расчёт метрик

### 32.1. Выполнение плана

```text
Выполнение плана = выполненные задачи / все задачи периода * 100
```

### 32.2. Доля просрочек

```text
Доля просрочек = просроченные задачи / все задачи * 100
```

### 32.3. Индекс загрузки сотрудника

Примерная формула:

```text
Индекс загрузки = активные задачи + задачи высокого приоритета * 1.5 + просроченные задачи * 2 + дедлайны в ближайшие 3 дня * 1.5
```

Интерпретация:

- 0–5 — нормальная;
- 6–10 — высокая;
- 11+ — перегрузка.

### 32.4. Эффективность отдела

```text
Эффективность = выполнение плана - доля просрочек - доля критических рисков
```

11. Открывает AI-помощника и спрашивает: «Кто перегружен на этой неделе?»
